# Sanctions Alert Service

A .NET 10 REST API for managing sanctions screening alerts. The service stores alerts in memory, enforces strict tenant isolation, and publishes alert transition events through an outbox-backed logging publisher.

## Tech Stack

- ASP.NET Core Web API
- .NET 10
- xUnit
- In-memory repository and outbox store

## How to Run

### Prerequisites

- .NET 10 SDK (for local development)
- Docker and Docker Compose (for containerized deployment)

### Option 1: Run Locally with .NET CLI

From the repository root:

```powershell
dotnet run --project src/SanctionsAlertService.Api
```

Swagger is enabled in the Development environment and opens automatically at:
- `http://localhost:5251/swagger` (HTTP)
- `https://localhost:7218/swagger` (HTTPS)

### Option 2: Run with Docker Compose

From the repository root:

```powershell
docker compose up --build
```

The API will be available at:
- `http://localhost:8080`
- Swagger UI: `http://localhost:8080/swagger`

To stop the container:

```powershell
docker compose down
```

### Run Tests

```powershell
dotnet test
```

## Tenant Isolation

Tenant identity is carried through the `X-Tenant-Id` request header.

Example:

```http
X-Tenant-Id: tenant-a
```

Requests without a tenant header are rejected with `400 Bad Request` and code `TENANT_REQUIRED`.

Isolation is enforced in two places:

1. `TenantMiddleware` resolves the tenant once per request and rejects unresolved tenants early.
2. All repository reads and writes are scoped by `tenantId`, so alerts from one tenant cannot be listed, fetched, or mutated by another tenant.

Threat model assumption: this assignment does not include authentication, so the service trusts the caller or upstream gateway to supply a valid tenant header. In production, tenant resolution should come from authenticated claims or a trusted API gateway instead of a raw caller-controlled header.

## API Design

Base route: `/api/v1/alerts`

### 1. Create alert

**Endpoint**

`POST /api/v1/alerts`

**Why**

Creates a new alert resource from an upstream screening system, so `POST` on the collection is the natural REST choice.

**Request body**

```json
{
  "transactionId": "txn-1001",
  "matchedEntityName": "ACME Trading LLC",
  "matchScore": 87,
  "assignedTo": "analyst-7"
}
```

**Behavior**

- Server generates `id`
- `status` starts as `OPEN`
- `createdAt` and `updatedAt` are generated by the service
- Duplicate `transactionId` within the same tenant returns `409 Conflict`

**Response**

- `201 Created`
- Returns the persisted alert
- Includes a `Location` header to `GET /api/v1/alerts/{id}`

### 2. List alerts

**Endpoint**

`GET /api/v1/alerts?status=OPEN&minMatchScore=80`

**Why**

Filtering is read-only and collection-oriented, so query string filters on `GET` are the simplest and most idiomatic design.

**Supported filters**

- tenant: required via `X-Tenant-Id`
- `status`: optional
- `minMatchScore`: optional, must be in `0..100`

**Response**

- `200 OK`
- Returns only alerts for the resolved tenant

### 3. Get alert by id

**Endpoint**

`GET /api/v1/alerts/{id}`

**Why**

Used by the create response and useful for reviewing a single alert. Tenant scoping still applies.

**Response**

- `200 OK` when found in the current tenant
- `404 Not Found` when the alert does not exist for the current tenant

### 4. Escalate an alert

**Endpoint**

`POST /api/v1/alerts/{id}/escalation`

**Why**

Escalation is modeled as a domain action rather than a generic partial update. `POST` to an action sub-resource makes the state transition explicit.

**Behavior**

- Valid only when current status is `OPEN`
- Transitions status to `ESCALATED`
- Publishes an `alert.escalated` domain event through the outbox

**Response**

- `200 OK` with the updated alert
- `409 Conflict` for invalid state transitions

### 5. Submit a decision

**Endpoint**

`POST /api/v1/alerts/{id}/decision`

**Why**

Decision submission is a write-once domain command with its own validation rules, so a dedicated action endpoint is clearer than a generic `PATCH`.

**Request body**

```json
{
  "decision": "CLEARED",
  "decisionNote": "False positive after manual review"
}
```

**Behavior**

- Accepts only `CLEARED` or `CONFIRMED_HIT`
- `decisionNote` is required
- Once decided, the alert cannot be decided again
- A second decision attempt returns `409 Conflict`
- Publishes an `alert.decided` domain event through the outbox

**Response**

- `200 OK` with the updated alert
- `409 Conflict` when the decision is retried

## Alert State Machine

Allowed transitions:

- `OPEN -> ESCALATED`
- `OPEN -> CLEARED`
- `OPEN -> CONFIRMED_HIT`
- `ESCALATED -> CLEARED`
- `ESCALATED -> CONFIRMED_HIT`

Disallowed transitions:

- `ESCALATED -> ESCALATED`
- `CLEARED -> *`
- `CONFIRMED_HIT -> *`

The decision operation is write-once. Once the alert reaches `CLEARED` or `CONFIRMED_HIT`, no further decisions or transitions are allowed.

## Event Publishing

When an alert is escalated or decided, the service emits a domain event.

Current design:

1. `AlertService` creates a domain event.
2. `IAlertUnitOfWork.SaveAsync(...)` persists the alert and outbox message(s) atomically in the same in-memory critical section.
3. API requests return successfully once persistence completes.
4. `OutboxDispatcherService` polls pending outbox messages.
5. `OutboxProcessor` deserializes and publishes events through `IAlertEventPublisher`.
6. `LoggingAlertEventPublisher` writes the serialized payload to application logs.

This is more production-friendly than publishing inline because it separates state changes from delivery retries and poison-message handling.

### Event Shapes

Decision event example:

```json
{
  "event": "alert.decided",
  "alertId": "8f9d8e3d-8a64-4c7a-8d48-53bff5393d27",
  "tenantId": "tenant-a",
  "decision": "CLEARED",
  "timestamp": "2025-01-01T12:00:00Z"
}
```

Escalation event example:

```json
{
  "event": "alert.escalated",
  "alertId": "8f9d8e3d-8a64-4c7a-8d48-53bff5393d27",
  "tenantId": "tenant-a",
  "outcome": "ESCALATED",
  "previousStatus": "OPEN",
  "timestamp": "2025-01-01T12:00:00Z"
}
```

## Error Handling

The API returns structured JSON errors with a status code, machine-readable code, message, timestamp, and path.

Examples:

- `400 TENANT_REQUIRED` when `X-Tenant-Id` is missing
- `400 VALIDATION_ERROR` for invalid input such as out-of-range scores
- `404 ALERT_NOT_FOUND` when an alert is not found in the tenant scope
- `409 DECISION_ALREADY_MADE` when a decision is retried
- `409 INVALID_STATE_TRANSITION` for unsupported transitions
- `409 TRANSACTION_ALREADY_EXISTS` when the tenant already has the same transaction id

## Design Decisions

### Layered architecture

The code follows controller -> service -> repository boundaries so infrastructure can be replaced without changing domain rules.

### In-memory storage with repository pattern

The assignment asks for no database, but the repository and unit-of-work abstractions keep storage concerns isolated and make a future database migration straightforward.

### Header-based tenant propagation

A request header is easy to test and keeps multi-tenancy explicit. The trade-off is that it is not secure by itself; in a real system, the tenant should be derived from authenticated identity.

### Action-style endpoints for transitions

Escalation and decision are business commands with strict state rules, so action endpoints communicate intent more clearly than broad update semantics.

### Outbox-backed event publishing

Even with in-memory storage, the service models reliable event publication separately from command handling. This demonstrates retry behavior, dead-letter handling, and cleaner evolution toward a real broker.

## Compromises / Deviations

- Storage is fully in memory, so data is lost on restart.
- Event publishing currently logs events instead of sending them to Kafka, RabbitMQ, NATS, or SQS.
- Tenant identity is not authenticated.
- The outbox polling model is intentionally simple for the assignment.
